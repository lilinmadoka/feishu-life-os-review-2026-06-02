# 产品规格：多端通用助理 / Feishu Life OS

## 1. 目标

用户每天面对学校事务、家教安排、学习任务、项目开发、聊天软件消息、邮箱通知、学习平台通知、临时想法、截图、通知栏信息、口头/文字约定。问题不是“缺少一个待办软件”，而是：信息分散、记录习惯不稳定、上下文容易丢、注意力被打断后无法回收。

本项目要做的是一个 **低摩擦捕获 + 自动整理 + 可追溯提醒 + 每日复盘** 系统。

一句话：任何事情先丢进一个入口，系统自动判断它是不是任务、日程、等待事项、提醒、笔记或需要确认的信息；再用飞书多维表格、飞书任务、飞书日历和机器人提醒承接后续执行。

## 2. 核心原则

### 2.1 先捕获，不要求用户立刻整理

用户注意力容易被打断，因此入口必须极低摩擦：

- 发一句话给飞书机器人。
- 转发聊天消息给机器人。
- 上传截图或语音，后续 OCR/ASR。
- 邮件转发到入口。
- 学习平台通知复制/转发到入口。

输入可以乱、可以短、可以只有“周五前数据库作业”。系统负责进入收件箱。

### 2.2 保留证据，不只保留结论

每个行动项都要保留 `evidence_text` 和 `source_ref`。当提醒出现时，用户需要知道：这件事从哪里来的、原话是什么、是否可能误解。

### 2.3 自动但不武断

高置信度事项可以直接生成行动项；低置信度事项进入 `needs_review`。不确定的时间、人名、任务类型不应该被“自信地误排”。

### 2.4 飞书是底座，不是唯一逻辑层

核心逻辑独立在后端服务里，飞书承担入口、数据展示、提醒和协作层。这样以后可以接微信、邮箱、学习平台、浏览器扩展或手机快捷指令，而不重写核心系统。

## 3. 用户场景

### 场景 A：家教临时约定

输入：

> 明天下午3点给学生小王补课，记得今晚把资料发给家长

系统输出：

1. 日程/事件：明天 15:00 给学生小王补课，领域 tutoring，预计 60 分钟。
2. 任务：今晚把资料发给家长，领域 tutoring/communication，低能量任务。
3. 原始证据保留在捕获表。

### 场景 B：聊天里出现学校 deadline

输入：

> 老师说周五前提交数据库作业，不要晚交

系统输出：

- 行动项：周五 23:59 前提交数据库作业。
- 领域：school。
- 优先级：根据当前日期自动给 P0-P2。
- 证据：原始聊天文字。

### 场景 C：脑子里的项目想法

输入：

> 今晚把项目 README 改完，明早让 Codex 接飞书 API

系统输出：

- 今晚处理 README。
- 明早处理 Codex/飞书 API 对接。
- 领域 project。

### 场景 D：等待别人回复

输入：

> 等张老师回复选课申请，后天上午如果没回就催一下

系统输出：

- waiting：等张老师回复选课申请。
- followup：后天上午催一下。

## 4. 信息类型

| 类型 | 说明 | 例子 | 承接方式 |
|---|---|---|---|
| task | 可执行任务 | 写作业、发资料、改 README | 飞书任务 / 多维表格 |
| event | 固定时间事件 | 家教、考试、会议 | 飞书日历 / 多维表格 |
| deadline | 截止事项 | 周五前提交 | 飞书任务 + 强提醒 |
| waiting | 等别人 | 等老师回复 | 每日复盘 + 跟进提醒 |
| followup | 到点提醒/催办 | 后天没回就催 | 飞书任务/机器人 |
| note | 暂存笔记 | 一个想法 | 多维表格收件箱 |
| habit | 重复事项 | 每天背单词 | 后续迭代 |

## 5. 优先级规则

- P0：已逾期、6 小时内截止、或含“马上/立刻/今晚必须”。
- P1：24 小时内截止、强约定、重要沟通。
- P2：3 天内截止、重要但可安排。
- P3：未来事项、低风险、暂存想法。

## 6. 默认每日节奏

### 早上 08:00

机器人推送：

- 今天必须处理。
- 昨天遗留/逾期。
- 今天的课程/家教/会议。
- 3 个最该清理的收件箱项目。

### 晚上 21:00

机器人推送：

- 今天没完成的 P0/P1。
- 明天早上的任务和日程。
- 需要发消息确认的事项。
- 未整理的新捕获。

### 中途提醒

只对 P0/P1 或明确截止时间事项强提醒，避免系统变成噪音源。

## 7. MVP 范围

已经实现：

- 文本捕获 API。
- 中文时间识别。
- 规则抽取任务/事件/等待事项。
- 自动领域、优先级、能量估计。
- SQLite 数据存储。
- 每日复盘生成。
- 飞书 API 适配层和 dry-run 同步记录。

Codex 需要继续做：

- 真正接入飞书机器人事件。
- 真正创建/更新飞书多维表格记录。
- 飞书任务和日历 payload 按当前开放平台调试通过。
- 截图 OCR、语音转写、邮件转发入口。
- 可选：LLM 抽取器替换/增强当前规则抽取器。

## 8. 非目标

- 不做复杂项目管理平台。
- 不要求用户每天严格维护表格。
- 不把所有聊天记录全量同步，避免隐私与噪音。
- 不在低置信度情况下自动创建强提醒。

## 9. 成功指标

1. 用户 80% 以上临时事项可以在 5 秒内捕获。
2. 每天早晚能看到清晰行动面板。
3. 重要 deadline、家教安排、项目约定漏掉次数显著下降。
4. 收件箱里低置信度事项不过夜，或至少在每日复盘中暴露。
5. 用户不需要打开多个软件回忆“到底有什么事”。
